Navigation API

실내 내비게이션 백엔드 API입니다. 건물·층·매장·POI 지도 데이터와 길찾기 그래프를 제공하고, 다익스트라 기반 최단 경로를 계산합니다. FastAPI + SQLAlchemy 2.0 ORM 구조로, Flutter 클라이언트가 소비하는 JSON 계약을 Pydantic 응답 모델로 고정합니다.

이 서비스는 읽기 중심 API입니다. 지도 데이터는 SVG 도면에서 변환된 JSON을 시드 스크립트로 적재하며, 런타임에는 조회와 경로 계산만 수행합니다. 현재 두 건물(thehyundai-seoul, test-center)의 1층 데이터가 기본 데이터셋으로 제공됩니다.

이 문서는 api/ 디렉토리의 백엔드만 다룹니다. Flutter 클라이언트(client/)와 SVG → 그래프 변환 파이프라인은 별도 문서를 참고하세요.

기술 스택

백엔드가 사용하는 기술을 계층별로 정리했습니다. 각 스택은 명확한 단일 책임을 가지며, 서로의 역할을 침범하지 않도록 설계되어 있습니다.

기술버전역할사용 위치
Python 3.12 런타임. match·type alias 등 최신 문법 사용 전체
FastAPI 0.115.* HTTP 라우팅, 의존성 주입(DI), OpenAPI/Swagger 자동 생성 app/main.py, app/routers/
SQLAlchemy ≥2.0 ORM. DeclarativeBase + Mapped 타입 기반 엔티티, Session 관리 app/models/, app/core/database.py
Pydantic 2.9.* HTTP 요청/응답 계약 검증. 테이블 모델로 사용하지 않음 app/schemas/
pydantic-settings ≥2.0 환경변수 기반 설정 (NAV_ prefix) app/core/config.py
SQLite 내장 개발용 파일 DB. 지도 좌표 배열은 JSON 컬럼으로 저장 api/data/navigation.db
Uvicorn 0.32.* ASGI 서버 실행 명령
pytest + httpx 최신 단위/통합 테스트. TestClient로 API 계약 검증 api/tests/

주요 기술 결정

  • SQLite 유지 — PostgreSQL 전환은 범위에서 제외. 개발 단계의 단일 파일 DB로 충분합니다.
  • Alembic 미도입 — 스키마가 자주 바뀌는 단계이므로 drop_all() → create_all() → seed로 매번 재생성합니다. 데이터 보존이 필요해지는 시점에 마이그레이션을 도입합니다.
  • Repository 패턴 미사용 — 단순 조회는 queries/의 Session 기반 함수로 처리합니다. 복잡한 쿼리 재사용이 실제로 생길 때만 도입을 검토합니다.
  • 동기(sync) 핸들러 — SQLAlchemy 동기 드라이버를 사용하므로 모든 핸들러는 def로 선언해 FastAPI가 스레드풀에서 실행하게 합니다. async def로 선언하면 블로킹 IO가 이벤트 루프를 멈춥니다.

Internals

시스템이 내부적으로 어떻게 동작하는지 — 계층 구조, 요청 처리, 경로 계산, 데이터 모델, 시드 흐름을 설명합니다.

내부 동작

아래 그림은 GET /buildings/{b}/floors/1F/route 요청 하나가 소켓에 도착해서 JSON 응답으로 나갈 때까지의 전체 관통 경로입니다. 바깥 상자는 안쪽 상자를 감싸며 실행됩니다(미들웨어 양파 구조). 요청은 바깥→안, 응답은 안→바깥으로 흐릅니다. 텍스트 원본은 docs/fastapi-request-flow.md에 Spring 대응 개념과 함께 정리되어 있습니다.

계층별 책임

계층하는 일하지 않는 일
routers/ URL/쿼리 파라미터 바인딩, Depends(get_db), response_model 선언, None→404 · ValueError→400 변환 ORM 조회, 비즈니스 계산, 응답 dict 조립
queries/ Session 기반 단순 조회, 기존 API JSON 형태로 dict 조립 HTTPException 등 FastAPI 타입 import
services/ 최단 경로 계산, geometry 방향 보정 — 규칙이 있는 로직만 HTTP 상태 코드 결정
schemas/ 외부 HTTP 응답 계약 검증 (Pydantic) 테이블 모델 역할, ORM 관계 직렬화
models/ 테이블 스키마, FK, 인덱스, ORM 관계 정의 API 응답 구조 결정
요청 생명주기

모든 요청은 동일한 Session 생명주기를 거칩니다. get_db()는 yield dependency로, 핸들러 실행 전에 Session을 열고 응답 전송 후 반드시 닫습니다. 

python# app/core/database.py
def get_db() -> Iterator[Session]:
    session = SessionLocal()
    try:
        yield session      # ← 여기서 핸들러가 실행된다
    except Exception:
        session.rollback()  # 예외 시 안전망
        raise
    finally:
        session.close()     # 응답 전송 후 항상 실행
  • Session은 요청마다 생성·종료되며 캐시하지 않습니다.
  • Engine은 프로세스 단위로 재사용됩니다 (모듈 전역).
  • 읽기 API뿐이므로 get_db()는 자동 commit을 하지 않습니다. 쓰기 유스케이스가 생기면 해당 Service가 명시적으로 session.commit()합니다.

N+1 방지 로딩 전략

건물 목록처럼 컬렉션을 함께 읽는 조회는 selectinload()로 배치 로딩합니다. 여러 컬렉션을 joinedload()로 한꺼번에 묶으면 곱집합으로 행이 폭증하므로 사용하지 않습니다. 

python# queries/building_queries.py — GET /buildings의 N+1 제거
buildings = session.scalars(
    select(Building).options(selectinload(Building.floors))
).all()
# → 쿼리 2번: buildings 전체 1번 + floors IN (...) 1번
최단 경로 계산

GET .../route만 Service를 거칩니다. 핵심 설계는 "DB는 두 번만 읽고, 탐색은 전부 메모리에서"입니다. 탐색 중 Edge.from_node 같은 ORM 관계를 따라가면 간선마다 lazy load 쿼리(N+1)가 발생하므로, 한 층의 Node·Edge 전체를 각각 한 번씩 조회해 id → 객체 딕셔너리로 변환한 뒤 다익스트라에 넘깁니다. 

textGET /buildings/{b}/floors/{f}/route?start_node_id=A&end_node_id=B
  │
  ▼
NavigationService(session).get_shortest_path(...)
  ├─ 1. Floor 조회 ─────────────── 없으면 None → Router가 404
  ├─ 2. 층의 Node 전체 조회 (쿼리 1번)
  ├─ 3. 층의 Edge 전체 조회 (쿼리 1번)
  ▼
domain.dijkstra.find_shortest_path(nodes, edges, start, end)
  ├─ 인접 리스트 구성 (bidirectional이면 역방향도 등록)
  ├─ heapq 기반 최단 거리 탐색
  ├─ 잘못된 노드 ID → ValueError → Router가 400
  └─ 연결 경로 없음 → None → path_found: false → Router가 404
  ▼
_build_path_points: 간선 geometry를 하나의 선으로 연결
  ├─ geometry 없는 간선 → 양 끝 노드 좌표로 보완
  ├─ 역방향 통과 간선 → 좌표 순서를 reverse()
  └─ 인접 간선의 공통 접점 좌표는 한 번만 포함
  ▼
RouteResponse (node_ids, edge_ids, path_points, total_distance_m)

domain/dijkstra.py는 순수 함수로, FastAPI도 SQLAlchemy Session도 모릅니다. ORM 객체가 가진 id, from_node_id, length_m 등의 속성만 duck-typing으로 사용하므로 단위 테스트에서 DB 없이 검증할 수 있습니다.

데이터 모델

8개 테이블로 구성됩니다. 지도 좌표 배열(geometry, polygon, coordinates)은 관계로 분해하지 않고 SQLite JSON 컬럼으로 유지합니다.

엔티티핵심 컬럼비고
Buildingid(PK), name, area_m2, footprint_local_m(JSON)건물 외곽 폴리곤 포함
Floorid(PK), building_id(FK), name, level(building_id, name) UNIQUE
Nodeid(PK), floor_id(FK), type, x_m, y_m, lat/lng길찾기 정점. lat/lng는 nullable
Edgeid(PK), from_node_id(FK), to_node_id(FK), length_m, bidirectional, geometry(JSON)Node를 두 번 참조 — foreign_keys 명시
Storeid(PK), floor_id(FK), name, centroid_x/y_m, entrance_node_id(FK, nullable), polygon(JSON)name에 인덱스 (검색용)
Poiid(PK), floor_id(FK), type, x_m, y_m, linked_node_id(FK, nullable)화장실·출구 등
FloorVectorMapfloor_id(PK+FK), coordinate_system(JSON), source(JSON)Floor당 0..1개 — SVG viewBox 메타데이터
MapFeature(id, floor_id) 복합 PK, kind, geometry_type, coordinates(JSON)렌더링용 벡터 도형

내부 컬럼명과 API 키는 다릅니다. DB의 from_node_id/to_node_id는 API에서 from/to로, centroid_x_m/centroid_y_m은 centroid_local_m: {x, y}로 노출됩니다. 이 변환은 queries/가 명시적으로 수행하며, Flutter 계약을 지키기 위해 from_attributes=True 같은 자동 매핑에 의존하지 않습니다.

노드 지도

시드된 실제 nodes·edges 데이터로 그린 인터랙티브 길찾기 그래프입니다. 층 버튼으로 지도를 전환하고, 체크박스로 표시할 노드 타입을 고르세요. 지도에서 노드를 클릭하면 출발점 → 도착점이 차례로 지정되고, 문서에 내장된 다익스트라가 서버와 동일한 규칙으로 최단 경로를 계산해 표시합니다.

시드 파이프라인

스키마 생성과 데이터 적재가 분리되어 있습니다. DDL 문자열은 존재하지 않으며, 테이블은 항상 ORM 모델 정의(Base.metadata)에서 생성됩니다. 

textpython -m scripts.reset_and_seed
  │
  ├─ reset_database()
  │    ├─ Base.metadata.drop_all(engine)     # 전체 테이블 삭제
  │    └─ Base.metadata.create_all(engine)   # ORM 정의대로 재생성
  │
  └─ seed_navigation(json_path)  × 데이터셋마다 반복
       ├─ navigation JSON 파싱 (건물 1 + 층 1 + nodes/edges/stores/pois)
       ├─ 간선 보완: geometry 없으면 양 끝 노드 좌표, length 없으면 polyline 길이 합
       ├─ vector_maps/ 디렉토리에서 건물·층이 일치하는 벡터 JSON 탐색
       │    └─ 일치 0개 → FileNotFoundError, 2개 이상 → ValueError
       └─ 전체를 하나의 트랜잭션으로 commit (실패 시 rollback)
  • seed_navigation()은 테이블을 지우지 않으므로, 여러 JSON에 대해 반복 호출하면 다건물 적재(append)가 됩니다.
  • 테스트는 session= 인자로 자신의 임시 DB Session을 주입해 같은 시드 코드를 재사용합니다.
  • FastAPI 서버 startup 이벤트에는 drop_all()/create_all()이 없습니다. 서버 재시작으로 데이터가 사라지는 사고를 막기 위한 규칙입니다.

모든 GET 엔드포인트는 Pydantic response_model로 응답 계약이 고정되어 있습니다. 전체 스키마는 서버 실행 후 /docs(Swagger UI)에서 확인할 수 있습니다.

GET /health 서버 생존 확인
GET /buildings 건물 요약 목록
GET /buildings/{building_id} 건물 상세 (footprint 포함)
GET /buildings/{building_id}/stores?q= 매장 이름 검색
GET /buildings/{building_id}/floors/{floor_name} 층 지도 데이터
GET /buildings/{building_id}/floors/{floor_name}/graph 길찾기 그래프
GET /buildings/{building_id}/floors/{floor_name}/route 최단 경로

자연어 질의 API

RAG 기반 자연어 질의는 후속 이슈로 예정되어 있으며, 현재는 요청 계약만 고정된 stub입니다.

POST /query/destination 목적지 질의 (예: "구찌 어디야?")
POST /query/info 장소 정보 질의 (예: "화장실 몇 층?") 
bashcurl -X POST "http://127.0.0.1:8000/query/destination" \
  -H "Content-Type: application/json" \
  -d '{"text": "구찌 어디야", "building_id": "thehyundai-seoul"}'

# → {"status": "stub", "query": "구찌 어디야", "result": null}
테스트

테스트는 세션 범위에서 임시 SQLite에 create_all → seed를 한 번 수행하고, 모든 Query/Service/API 테스트가 이 시드 DB를 공유합니다. API 테스트는 dependency_overrides로 core.database.get_db만 테스트 Session으로 교체합니다. 

powershell# api/ 디렉토리에서 전체 테스트 실행
python -m pytest tests -q
# 44 passed
범위검증 내용
unit/test_dijkstra.py최단 거리 선택, 단방향 간선, 동일 노드, 음수 가중치 거부
unit/test_navigation_service.py정방향/역방향 geometry, geometry 누락 보완, 접점 중복 제거, 경로 없음

integration/test_seed.py시드 후 ORM 관계(층-노드-간선-벡터지도) 유효성
integration/test_queries.pyQuery 함수의 식별자·JSON 키 구조
integration/test_*_api.py엔드포인트별 정상 응답, 대표 JSON 키, 400/404 경계
규칙과 관례
  • Flutter 계약 보존 — URL, HTTP 상태 코드, 응답 JSON 키(from, to, path_points, centroid_local_m 등)는 변경하지 않습니다.
  • Pydantic ≠ 테이블 모델 — schemas/는 HTTP 계약 전용입니다. ORM 관계 객체를 그대로 직렬화하지 않습니다.
  • DB 초기화는 CLI에서만 — 서버 startup에서 drop_all()/create_all()을 호출하지 않습니다.
  • 핸들러는 동기(def) — 동기 DB 드라이버를 쓰는 동안 async def 핸들러를 만들지 않습니다.
  • 설명·커밋 메시지는 한국어, 코드·식별자는 영어 — 저장소 공통 규칙 (커밋 메시지는 영어 한 줄 요약도 허용).
