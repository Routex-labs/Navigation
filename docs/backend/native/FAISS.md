# AI 자연어 질의 (FAISS + 형태소) — 구현 설계

경량 텍스트 매칭([query.md](query.md))의 후속 단계. `query.md` 2절·9절에서 "후속"으로 남겨 둔
`② 형태소 정규화(Kiwi) → ③ 사전학습 임베딩(FAISS)` 을 실제로 구현하기 위한 설계 문서다.
구현 전에 **아키텍처·형태소기·벡터화 방식·실패 조건·검증 기준**을 먼저 확정한다.

> **구현 완료 상태.** 5절 3안 비교 후 **C(하이브리드)**로 확정·구현했다(11절). 임베딩 모델은
> `ko-sroberta-multitask`, 엔드포인트는 `POST /query/ai`, 임계값 0.50(실데이터 튜닝).
> 형태소(Kiwi)는 이번 범위에서 보류 — 문장 임베딩만으로 자연어 매칭이 실데이터에서 동작함을 확인.

## 1. 해결할 문제

- 기존 경량 매칭은 **이름·카테고리·동의어의 문자열 일치**만 한다. 사전에 없는 표현은 못 잡는다.
  - "밥 먹을 데" → 음식점? 못 잡음(동의어 사전에 없음).
  - "애들 신발" → 아동 슈즈 매장? 못 잡음.
  - "화장실 급해" → 꼬리 제거 규칙 밖이라 약함.
- 동의어 사전을 무한정 늘리는 방식은 유지보수가 깨진다. **개념 단위 의미 검색**이 필요하다.
- 완료 조건(Definition of Done):
  - 사용자가 **AI 쿼리 버튼**으로 자연어를 넣으면, 사전에 없는 표현도 **의미가 가장 가까운 매장 1건**과 그 **입구 노드**를 돌려준다(온디바이스 경로용).
  - 경량 매칭의 응답 계약(`dto/query.py`)을 깨지 않는다 — 클라이언트 파싱 재사용.
  - 실패 조건(임계값 미달·건물 없음·입구 노드 없음)이 명시된 상태로 처리된다.

## 2. 범위

- **1차 목표 — 매장 안내.** 자연어 질의 → 매장 1건 + `entrance_node_id`. 경로 계산은 여전히 클라이언트(Dijkstra).
- **이번 범위 — AI 질의 경로 신설.** 기존 경량 매칭은 그대로 두고(상단 일반 검색), **별도 경로**로 형태소+FAISS 의미 검색을 얹는다. 두 경로는 응답 계약을 공유한다.
- **범위 밖 — 상품 광고·연관 매장 추천.** "이 근처 세일하는 곳", "MLB랑 비슷한 매장" 같은 2차 기능은 후속. 지금은 이음새(임베딩 인덱스)만 만들어 두고 구현은 미룬다.
- **범위 밖 — 경로 계산.** 서버는 도착 노드까지만. 최단 경로는 클라이언트.
- **범위 밖 — 대화형 LLM·생성 응답.** 이번 건 **검색(retrieval)** 이지 생성이 아니다. 문장 생성·요약 없음.
- **1차 대상 데이터 — B2 매장.** `query.md`와 동일하게 지하 2층 데이터로 먼저 구현·검증하고, 나머지 층은 같은 파이프라인으로 커버된다.

## 3. 아키텍처 — 두 경로의 공존

```
[상단 일반 검색]  ──▶  경량 매칭(_rank, 동의어 사전)      ── 기존, 그대로 유지
   빠름·정확 이름/동의어

[AI 쿼리 버튼]   ──▶  형태소 정규화(Kiwi) ──▶ FAISS 의미 검색  ── 이번 신설
   자연어·개념                                    │
                                                  ▼
                              둘 다 QueryMatch(같은 응답 계약)로 수렴
```

- **엔드포인트**: 신설 `POST /query/ai` (또는 기존 `/query/destination`에 `mode` 플래그). 응답은 `DestinationResponse`를 그대로 사용해 클라이언트 파싱을 재사용한다. 확정은 5절 선택 후.
- **경량 매칭을 지우지 않는 이유**: 정확한 브랜드명("MLB")은 임베딩 유사도보다 문자열 일치가 더 정확·빠르고 오매칭이 없다. 상단 검색의 즉답성은 경량이 담당하고, AI 버튼은 "말로 푸는" 질의를 담당한다. 역할 분리.

## 4. 형태소 분석기 — Kiwi 확정

**`kiwipiepy` (Kiwi) 로 확정.** 질의·매장명 양쪽을 형태소로 쪼개 정규화·토큰화에 쓴다.

- **왜 형태소가 필요한가**: 한국어는 조사·어미가 붙어 문자열이 흔들린다("화장실이", "화장실은", "화장실 어디"). 형태소로 **체언(명사)만 추출**하면 질의가 안정된다. 임베딩·TF-IDF 어느 쪽을 골라도 전처리로 품질이 오른다.
- **역할**:
  1. 질의 정규화 — 명사/고유명사만 남기고 조사·어미·의문형 제거(현재 `_TAILS` 하드코딩 규칙을 형태소 기반으로 일반화).
  2. 토큰화 — 벡터화 대상 텍스트(매장명+카테고리)를 형태소 단위로 분해.
  3. 사용자 사전 — `add_user_word`로 브랜드 신조어("마뗑킴", "아르켓")를 등록해 오분해 방지. 기존 `query_synonyms.json`과 연결.
- **선택 근거**:
  - 순수 pip 휠 설치, **JVM 불필요** — Windows·Docker·macOS 셋업이 동일하고 가볍다.
  - C++ 기반이라 빠르고, 신조어·띄어쓰기 교정에 강하다(브랜드명 다수인 이 데이터에 유리).
  - `query.md` 2절이 이미 "형태소 정규화(Kiwi 등)"로 지목 — 문서 일관성.

### 미선택 — 이유만 기록

- **KoNLPy(Okt/Komoran) / Mecab** — 성숙하지만 **JVM(jpype) 또는 네이티브 C 빌드 + 사전(mecab-ko-dic)** 이 필요하다. Windows·Docker 빌드 단계가 늘고, 개발자가 양 OS에 있는 이 프로젝트에선 셋업 마찰이 크다. Kiwi로 충분해 채택하지 않는다.

## 5. 벡터화 3안 비교 — **여기서 하나 확정**

FAISS 인덱스에 넣을 벡터를 무엇으로 만들지가 의존성·이미지 크기·"진짜 의미검색" 수준을 정한다.
코퍼스 규모는 건물당 매장 수백~수천 건으로 작다(→ FAISS 인덱스는 어느 안이든 `IndexFlat` 브루트포스로 충분, IVF/HNSW 불필요).

| 항목 | A. 문장 임베딩(SBERT) | B. 형태소 TF-IDF + FAISS | C. 하이브리드(A+경량) |
|---|---|---|---|
| **핵심** | 한국어 SBERT로 문장을 밀집 벡터화 → 코사인 유사도 | Kiwi 토큰 → TF-IDF, LSA로 밀집화 → FAISS | 경량/형태소로 1차 정확·동의어, 미달 시 A로 2차 |
| **의미 이해** | ★★★ "밥 먹을 곳"→음식점 등 개념 매칭 | ★ 같은 형태소 공유해야 매칭(개념 X) | ★★★ (A와 동일) + 정확매칭 보정 |
| **신규 의존성** | `sentence-transformers`(+torch), `faiss-cpu` | `scikit-learn`, `faiss-cpu` | A와 동일 |
| **이미지/디스크** | 무거움(torch ~수백MB, 모델 120~500MB) | 가벼움(torch 없음, 수십MB) | 무거움(A와 동일) |
| **질의 지연(CPU)** | 임베딩 ~10~50ms + 검색 | ~수ms | 1차 히트면 ~수ms, 2차만 임베딩 |
| **콜드스타트** | 모델 로드 수 초, 오프라인 캐시 필요 | 거의 없음 | A와 동일(2차 경로 로드) |
| **오매칭 위험** | 정확한 이름도 유사도에 맡겨 흔들릴 수 있음 | 낮음(어휘 일치 기반) | 낮음(정확 이름은 1차가 확정) |
| **구현·문서 복잡도** | 중 | 중(LSA 차원축소 단계) | 높음(두 경로 관리) |
| **"진짜 자연어" 부합** | 높음 | 낮음(사실상 나은 어휘검색) | 가장 높음 |

- **추천: C(하이브리드).** 이유:
  - 정확한 브랜드명은 **이미 만들어 둔 경량 매칭**이 확정하므로(오매칭 0), 임베딩은 "말로 푸는" 질의만 담당 → A 단독의 이름 오매칭 위험이 사라진다.
  - 추가 비용은 A 대비 크지 않다. 1차 경로가 이미 존재(`query_search._rank`)하므로, 그 뒤에 임베딩 fallback만 붙이면 된다.
  - 3절 아키텍처(경량=상단검색, AI=버튼)와 자연스럽게 맞는다.
- **차선: A(문장 임베딩 단독).** AI 버튼 경로를 처음부터 의미검색으로 단순하게 갈 때. 이름 오매칭은 임계값·정확일치 가드로 보완.
- **비추천 상황의 B.** "진짜 자연어"라는 이번 목표엔 개념 이해가 약해 부족하다. torch를 절대 넣기 싫은 제약이 우선일 때만.

> 임베딩 모델 후보(A/C 채택 시 별도 확정): CPU·경량 우선이면 `intfloat/multilingual-e5-small`(384d) 또는
> 한국어 특화 `jhgan/ko-sroberta-multitask`(768d). 데모 지연·이미지 크기를 보고 소형부터.

## 6. 인덱싱 파이프라인 (A/C 채택 시)

- **인덱스 문서 단위**: 매장 1건 = `name + category + subcategory`(+동의어)로 만든 텍스트. Kiwi 토큰화 후 임베딩.
- **빌드 시점**: 시드/서버 기동 시 건물별로 1회 빌드해 메모리 상주(코퍼스가 작아 부담 없음). DB가 시드로 초기화되므로 **기동 시 재빌드**가 가장 단순 — 인덱스 파일 영속화는 후속.
- **정규화**: 코사인 유사도 → 임베딩 L2 정규화 후 `IndexFlatIP`.
- **매핑**: FAISS 벡터 행 → `store.id` 역참조 테이블 유지(검색 결과를 `Store`로 되돌린다).

## 7. 질의 처리 흐름 (C 기준)

1. 요청 `{ text, building_id, current_floor_id? }` 수신.
2. Kiwi로 질의 정규화(명사 추출, 조사·의문형 제거).
3. **1차**: 경량 `_rank`로 정확 이름/동의어/카테고리 매칭 시도. 히트하면 그 매장 확정.
4. **2차**: 1차 미스면 건물 인덱스에서 FAISS Top-k 검색 → 최상위 1건.
5. `current_floor_id`가 있으면 해당 층으로 후보를 한정(경량·FAISS 양쪽).
6. `QueryMatch` 조립(입구 노드·좌표·층명) 후 `DestinationResponse`로 반환.

## 8. 실패 조건 (정상 동작보다 먼저 정의)

| 상황 | 처리 |
|---|---|
| `building_id`가 없는 건물 | `404` (기존 계약과 동일하게 라우터가 번역) |
| 1차·2차 모두 미스 | `200` + `{"status":"no_match","query":…,"match":null}` (에러 아님) |
| FAISS 최상위 유사도가 **임계값 미달** | `no_match`로 처리(엉뚱한 매장 반환 금지). 임계값은 실데이터로 튜닝하고 문서에 기록. |
| 매칭됐으나 `entrance_node_id`가 `null` | `match`는 주되 `status:"ok_no_route"`로 구분(경로 불가 인지). |
| `text`가 빈 문자열/공백 | `422`(요청 검증) — 경량 매칭과 일관. |
| 임베딩 모델 로드 실패 | 서버 기동은 유지하되 AI 경로만 비활성(경량은 계속 동작). 로그로 명시. |
| 동점·비결정성 | 유사도 동점은 `(level, store.id)` 등 **결정적 정렬**로 1건 확정(테스트 재현성). |

## 9. 응답 계약 (기존 유지)

- `dto/query.py`의 `QueryMatch` / `DestinationResponse`를 **그대로 사용**한다. 새 필드 없이도 AI 경로가 매장 안내를 만족한다.
- 유사도 점수를 클라이언트에 노출할지는 후속 결정(디버그용으로만 로그에 남기고 계약엔 넣지 않는 편을 기본으로).

- **확정 — 유사도 임계값**: `SIMILARITY_THRESHOLD = 0.50`. thehyundai-seoul 실데이터(매장 1531건, 질의 27개)로 튜닝. 근거는 아래 11-1.
- **확정 — 역할 분리**: 경량 매칭 = 상단 일반 검색(유지), FAISS = AI 쿼리 버튼(신설).
- **확정 — 1차 목표**: 매장 안내(매장 1건 + 입구 노드). 상품 광고·연관 매장은 후속.
- **보류 — 형태소기(Kiwi)**: 이번 FAISS 구현에는 미포함. 질의 정규화 개선은 별도 단계로 남긴다(문장 임베딩만으로 자연어 매칭이 실데이터에서 동작 확인).
- **미결 — 임베딩 인덱스 영속화**: 현재 기동 시 재빌드(콜드스타트 시 모델 로드+인코딩). 필요 시 파일 저장/로드.

### 11-1. 임계값 0.50을 더 낮추지 않은 이유 — 마진이 0.006뿐

thehyundai-seoul 27개 질의 평가에서, **0.50 미만이라 놓치지만 사실 맞는** 질의가 셋 있다.

| 질의 | top 점수 | 최상위 후보 | 문제 |
|---|---|---|---|
| `현금 뽑을 데` | 0.464 | ATM (하나은행) | ATM이 top인데 미달로 놓침 |
| `가방 보러 왔어` | 0.453 | 타임파리(잡화·액세서리) | 관련 매장이나 미달 |
| `에스컬레이터` | 0.447 | 리모와(명품) — 엉뚱 | 에스컬레이터가 store 인덱스에 없음(데이터 공백) |

그런데 무의미 질의 점수가 바로 아래 붙어 있다: `asdfqwerzxcv` **0.458**, `ㅋㅋㅋㅋㅋ` 0.437.

**유효한 ATM(0.464)과 무의미(0.458)의 간격이 0.006뿐이다.** 임계값을 0.46으로 낮추면
ATM은 잡지만 `asdfqwerzxcv`도 통과해 엉뚱한 매장을 답으로 내보낸다. 길찾기에서는
"틀린 매장 안내" > "다시 말해 주세요" 가 더 나쁘므로, 0.50은 **정밀도 우선** 선택이다.

즉 이 값은 재현율을 몰라서 높게 잡은 게 아니라, 위 마진 때문에 **절대 임계값 하나로는
더 못 내리는** 것이다. 재현율을 올리려면 임계값이 아니라 다음을 손대야 한다.

1. **임베딩 텍스트 보강** — 매장에 설명·동의어를 붙여 임베딩(예: ATM → "ATM 현금 인출 은행"). 이름만으로는 신호가 약하다.
2. **형태소(Kiwi) 정규화** — 질의 노이즈 제거로 점수 안정화(위 "보류" 항목과 연결).
3. **POI 시설 처리** — 에스컬레이터처럼 store에 없는 시설은 POI로 별도 검색(9절 이중표현 정리와 연결).
4. **마진/상대 기준** — 절대 임계값 대신 top1-top2 간격 등으로 무의미 질의를 거른다.

> 질의별 상위 3건 점수 나열은 [`backend/notebooks/faiss_eval.ipynb`](../../../backend/notebooks/faiss_eval.ipynb)에서 재생성한다.
