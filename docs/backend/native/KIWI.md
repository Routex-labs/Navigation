# 형태소 정규화 (Kiwi) — 구현 설계

[FAISS.md](FAISS.md) 4절에서 "형태소 분석기는 Kiwi로 확정하되 이번 범위는 보류"로 남긴 항목을
실제로 구현하기 위한 설계 문서다. 구현 전에 **적용 지점·실패 조건·검증 기준**을 먼저 확정한다.

> **구현 완료 상태.** 4절 결정 항목 확정 후 구현했다. `repositories/query_morph.py` 신설,
> `query_search._normalize_query`가 2단계(꼬리 제거 → 형태소)로 바뀌었다. 임베딩 경로는 손대지 않았다.

## 1. 해결할 문제

경량 1차 매칭(`repositories/query_search.py`)의 질의 정규화가 **하드코딩된 꼬리 목록**이다.

```python
_TAILS = ("몇 층이야", "몇층이야", "몇 층", "몇층", "어디야", "어디", "위치", "알려줘")
```

- 목록에 없는 조사·어미는 못 벗긴다: `"화장실이"`, `"스타벅스는"`, `"엘리베이터까지"`, `"화장실 급해"`.
- 꼬리를 **최대 1개만** 제거한다: `"화장실 어디야"`는 `"어디야"`만 떨어져 `"화장실"`이 되지만,
  `"화장실이 어디야"`는 `"화장실이"`가 남아 `_tier`의 정확 일치(tier 0)·카테고리 일치(tier 1)에서 모두 빠진다.
- 부분 일치(tier 2)가 `q in name` 방향이라 **질의가 더 길면 매칭이 끊긴다**. 조사가 붙는 순간
  `"화장실이" in "화장실"` 은 거짓이다. 즉 조사 하나로 1차가 통째로 비고 2차(임베딩)로 넘어간다.
- 2차로 넘어간다는 건 **torch 로드 + 임베딩 인코딩 비용**을 조사 하나 때문에 치른다는 뜻이다.
  게다가 `query_search.match_destination`·`match_info`(일반 검색·정보 질의)에는 2차가 없어서,
  그쪽은 그냥 `no_match`가 된다.

완료 조건(Definition of Done):

- 조사·어미가 붙은 자연어 질의가 **1차 경량 매칭에서** 잡힌다(임베딩 없이).
- 기존 응답 계약(`dto/query.py`)과 기존 테스트가 그대로 통과한다.
- Kiwi가 없는 환경에서도 서버는 기동하고, 기존 `_TAILS` 규칙으로 동작한다(품질만 하락).

## 2. 범위

- **이번 범위** — `query_search._normalize_query` 를 형태소 기반으로 교체. 브랜드 사용자 사전 등록.
  `match_destination` / `match_info` / `match_ai_destination` 1차가 모두 이득을 본다(공유 함수).
- **범위 밖** — 임베딩 문서 텍스트(`query_semantic._document_text`) 변경. 인덱스 텍스트를 바꾸면
  FAISS 임계값 0.50의 튜닝 근거가 무효가 된다(FAISS.md 8절). 별도 작업으로 재평가와 함께 한다.
- **범위 밖** — 임계값·리랭커·동의어 사전 확장.
- **범위 밖** — 클라이언트 변경. 서버 내부 매칭 품질 개선이라 API 계약이 그대로다.

## 3. 설계

### 3.1 적용 지점

```
질의 텍스트
   │
   ▼
_normalize_query()  ◀── 여기만 교체 (Kiwi 명사/어근 추출)
   │
   ├─▶ 동의어 사전(canon)          ── 그대로
   └─▶ _tier() 매칭                ── 그대로
```

`_tier`·`_rank`·`_load_stores`·DTO·라우터는 건드리지 않는다. 정규화 함수 하나의 입출력만 바뀐다.

### 3.2 정규화 규칙 — 실제 분석 결과로 두 번 수정됨

초안은 "명사만 뽑아 공백으로 이어 붙인다"였다. 실제 Kiwi 출력을 찍어 보고 **두 군데가 틀렸다.**

**(가) 토큰을 이어 붙이면 안 된다 → 원문 스팬에서 제거만 한다.**

```
"가게A 어디야" → [('가게',NNG,0,2), ('A',SL,2,1), ('어디',NP,4,2), ('이',VCP), ('야',EF)]
```

공백으로 join하면 `"가게 A"`가 되어 이름 정확 일치(tier 0)가 깨진다. 공백 없이 join하면
`"TAX REFUND"` → `"TAXREFUND"`로 또 깨진다. **원문의 문자 위치를 유지한 채 제거 대상 스팬만
공백으로 덮고 공백을 정리**하는 방식만이 둘 다 살린다.

**(나) 명사 허용 목록으로는 "몇 층이야"를 못 뗀다 → 꼬리 제거를 앞단에 유지한다.**

```
"화장실 몇 층이야" → [('화장실',NNG), ('몇',MM), ('층',NNG), ('이',VCP), ('야',EF)]
```

`"층"`은 일반명사(`NNG`)라 명사 필터를 통과한다 → `"화장실 층"`. 기존 `_TAILS` 꼬리 제거를
**폴백이 아니라 1단계 전처리로도** 남겨 `"몇 층이야"`를 먼저 떼고, 그다음 형태소가 남은 조사를 뗀다.

최종 규칙:

1. `_strip_tail()` — 기존 `_TAILS` 꼬리를 최대 1개 제거(`"화장실 몇 층이야"` → `"화장실"`).
2. `query_morph.normalize()` — Kiwi로 분석해 **제거 대상 태그의 스팬만** 지운다.
   제거: `J*`(조사) `E*`(어미) `V*`(용언) `MM`(관형사 "몇") `MA*`(부사) `NP`(대명사 "어디")
   `XSV`/`XSA`(용언파생접미사). 그 외는 남긴다.
3. **허용 목록이 아니라 제거 목록인 이유**: 모르는 태그는 남기는 쪽이 기존 동작에 가깝다.
   실제로 `"B1 주차"`의 `"주차"`는 `NNB`(의존명사), `"애들"`의 `"들"`은 `XSN`으로 분석된다 —
   명사 허용 목록(`NNG`/`NNP`)만 썼다면 둘 다 잘려 나갔다.
4. **남는 게 없으면 1단계 결과로 폴백.** 분석기가 다 날려도 지금과 같은 동작으로 떨어진다.
5. 결과는 소문자화(`_norm`)해 기존 비교 규칙과 동일하게 맞춘다.

### 3.3 브랜드 오분해 방지

`resources/query_synonyms.json`의 **키·값 양쪽을 사용자 사전(`add_user_word`)에 등록**한다.
`"마뗑킴"`, `"아르켓"`, `"노스페이스"` 같은 신조어가 `"마" + "뗑킴"` 으로 쪼개지는 걸 막는다.
사전 파일이 이미 로드되어 있으므로 새 리소스 파일을 만들지 않는다.

### 3.4 실패 시 폴백

Kiwi는 **선택 의존성**으로 다룬다. `query_semantic._get_model()`과 같은 패턴이다.

- `import kiwipiepy` 실패 / 인스턴스 생성 실패 → 플래그를 세우고 **기존 `_TAILS` 규칙으로 폴백**.
- 분석 중 예외 → 해당 질의만 원본 텍스트로 폴백(서버는 죽지 않음).
- 인스턴스는 지연 로드 싱글턴 + 락. 요청마다 새로 만들지 않는다.

| # | 항목 | 결정 | 근거 |
|---|---|---|---|
| Q1 | Kiwi 의존성 등급 | **필수 + 폴백 유지** — `requirements.txt`에 `kiwipiepy==0.23.*` | 환경마다 매칭 동작이 달라지면 버그 재현이 안 된다. 순수 pip 휠이라 JVM·빌드 단계가 없어 양 OS 셋업이 동일하다. 그래도 import 실패 시 폴백은 남긴다. |

| Q3 | tier 2(부분 일치) 방향 | **이번엔 그대로** | 정규화 함수 하나만 바꿔 변경 범위·회귀 위험을 최소화한다. 매칭 규칙 변경은 효과를 따로 측정해 별도 작업으로. |

## 5. 실패 조건 — 먼저 짚어 둔다

- **한글+영문 붙은 이름**: `"가게A"`가 `"가게 A"`로 벌어지면 tier 0이 깨진다 → 3.2 (가) 스팬 방식으로 방어.
- **영문·숫자 시설명**: `"TAX REFUND"`(`SL SL`), `"B1 주차"`(`SL SN NNB`)가 보존돼야 한다.
  제거 목록 방식이라 통과하지만, 목록을 늘릴 때 다시 깨질 수 있다 → 회귀 테스트로 고정했다.
- **한 글자 매장명**: 명사 추출 후 한 글자만 남으면 tier 2 부분 일치가 과매칭될 수 있다(미해결, Q3 범위 밖).
- **동의어 키가 명사로 안 잡히는 경우**: `"스벅"`, `"엘베"`, `"마뗑킴"` → 3.3 사용자 사전으로 방어.
- **Kiwi 미설치 환경**: 폴백이 동작하는지 monkeypatch로 재현해 테스트했다.
- **성능**: Kiwi 인스턴스 최초 생성 비용(수백 ms). 지연 로드 싱글턴이라 첫 질의만 부담한다.
- **여전히 1차에서 안 잡히는 것**: `"밥 먹을 데"` → `"밥 데"`, `"애들 신발"` → `"애들 신발"`.
  개념 검색이라 형태소로는 안 되고 임베딩(2차)의 몫이다. 2차에는 **원문 그대로** 넘어가므로
  이 변경이 임베딩 결과에 영향을 주지 않는다.

## 7. 남은 작업

- 동의어 사전(`query_synonyms.json`)에 없는 브랜드는 여전히 오분해 가능 — 실데이터로 확인 후 보강.
- tier 2 매칭 방향(Q3): `"여자 화장실"` 같은 다중 명사 질의는 아직 1차에서 안 잡힌다.
- 임베딩 문서 텍스트에도 형태소를 적용할지는 FAISS 임계값 재튜닝과 함께 판단.
