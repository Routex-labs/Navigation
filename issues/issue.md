# ISSUE

Navigation 프로젝트의 **마일스톤별 GitHub 이슈 초안과 설명**을 모아두는 문서다.
실제 GitHub 이슈를 만들기 전에, 어떤 마일스톤에 어떤 작업을 넣을지와 그 이유를 이 파일에서 먼저 정리한다.

> 작성 규칙
> - 마일스톤 단위로 섹션을 나눈다.
> - 이슈 식별자는 `ISSUE-NNN` 형식을 사용한다.
> - 상태는 `Draft`, `Ready`, `Created`, `Done` 중 하나로 기록한다.
> - GitHub 이슈로 생성한 뒤에는 `GitHub` 칸에 이슈 번호를 적는다.
> - 각 이슈의 `설명`은 GitHub 이슈 본문에 그대로 옮길 수 있게 작성한다.

## M0 · 프로젝트 기반 정리

### ISSUE-001 · 프로젝트 방향과 사용자 시나리오 정의

- **상태**: Draft
- **컴포넌트**: planning
- **GitHub**: -

#### 설명

Navigation 프로젝트가 해결하려는 이동 문제를 한 문장으로 고정한다.
경진대회 발표에서 심사자가 바로 이해할 수 있도록 사용자, 상황, 불편함, 해결 방식을 분명히 정리한다.

#### 작업 내용

- 핵심 사용자 1명을 정의한다.
- 사용자가 겪는 이동/길찾기 문제를 구체적인 상황으로 적는다.
- Navigation이 제공할 차별점을 1~2개로 좁힌다.
- README와 `docs/navigation-overview.md`에 결정 내용을 반영한다.

#### 수용 기준

- 프로젝트 목적이 한 문장으로 정리되어 있다.
- 발표 첫 화면에 넣을 수 있는 사용자 시나리오가 작성되어 있다.
- 이후 작성할 기능 우선순위가 이 시나리오와 연결될 수 있다.

### ISSUE-002 · Navigation 저장소 기본 문서 구조 정리

- **상태**: Draft
- **컴포넌트**: docs
- **GitHub**: -

#### 설명

개발 전에 저장소의 문서 구조를 고정해 팀원이 어디에 무엇을 적어야 하는지 헷갈리지 않게 한다.
README, 버전, 조사 노트, 자동화 프롬프트, 이슈 초안의 역할을 분리한다.

#### 작업 내용

- README의 디렉토리 구조를 최신 상태로 유지한다.
- `VERSION.md`의 컴포넌트와 이미지 태그가 Navigation 기준인지 확인한다.
- 구현 방향이 정해지기 전까지는 별도 로드맵 문서를 만들지 않는다.
- `docs/research-notes.md`에 조사 항목을 추가한다.

#### 수용 기준

- 새로 온 팀원이 README만 보고 문서 위치를 이해할 수 있다.
- 이전 프로젝트 전용 표현이 남아 있지 않다.
- 마일스톤 이슈 초안이 이 파일에서 계속 관리된다.

## M1 · Walking Skeleton

### ISSUE-003 · 더미 경로 API 계약 정의

- **상태**: Draft
- **컴포넌트**: api / routing
- **GitHub**: -

#### 설명

지도 API나 실제 경로 알고리즘이 완성되기 전에 프론트엔드와 백엔드가 같은 응답 형식을 바라볼 수 있도록 더미 경로 API 계약을 먼저 정한다.
이 계약이 잡혀야 UI와 라우팅 구현을 병렬로 진행할 수 있다.

#### 작업 내용

- 출발지, 도착지, 조건 옵션을 담는 요청 형식을 정의한다.
- 추천 경로, 대안 경로, 거리, 예상 시간, 추천 이유를 담는 응답 형식을 정의한다.
- 더미 응답 예시를 문서 또는 mock 파일로 남긴다.

#### 수용 기준

- 프론트엔드가 실제 API 없이도 mock 응답으로 화면을 만들 수 있다.
- API 응답에 추천 이유와 비교 지표가 포함되어 있다.
- 잘못된 입력에 대한 에러 형식이 정리되어 있다.

### ISSUE-004 · 데모 화면에서 더미 경로 표시

- **상태**: Draft
- **컴포넌트**: client / demo
- **GitHub**: -

#### 설명

사용자가 출발지와 도착지를 입력하면 더미 경로 결과가 화면에 표시되는 최소 데모를 만든다.
초기 목표는 완성도가 아니라 전체 흐름이 끊기지 않는지 확인하는 것이다.

#### 작업 내용

- 출발지/도착지 입력 UI를 만든다.
- 더미 API 응답을 화면에 표시한다.
- 추천 경로와 대안 경로를 구분해서 보여준다.
- 거리, 예상 시간, 추천 이유를 함께 보여준다.

#### 수용 기준

- 로컬에서 화면을 열고 한 번의 입력으로 경로 결과를 확인할 수 있다.
- API가 아직 없어도 mock 데이터로 데모가 가능하다.
- 발표용 시나리오로 확장할 수 있는 화면 흐름이 잡혀 있다.

## M2 · 데이터와 기본 라우팅

### ISSUE-005 · 데이터 소스 후보 조사 및 샘플 데이터 확정

- **상태**: Draft
- **컴포넌트**: data / routing
- **GitHub**: -

#### 설명

Navigation의 경로 계산이나 추천에 사용할 데이터 소스를 조사하고, 외부 API 없이도 데모가 가능한 샘플 데이터를 확정한다.
외부 서비스 제한 때문에 발표가 막히지 않도록 fallback 데이터를 준비하는 것이 핵심이다.

#### 작업 내용

- 지도 API, 공공데이터, 직접 제작 샘플 데이터 후보를 비교한다.
- 최소 5개 이상의 데모 시나리오를 정한다.
- 각 시나리오에 필요한 출발지, 도착지, 경로 후보, 비교 지표를 정리한다.
- `docs/research-notes.md`에 조사 근거를 남긴다.

#### 수용 기준

- 외부 API가 없어도 실행 가능한 샘플 데이터가 있다.
- 데이터 선택 이유가 문서에 설명되어 있다.
- 이후 확정될 구현 순서와 연결되는 다음 구현 이슈를 만들 수 있다.
